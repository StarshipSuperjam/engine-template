## Rule

<State the standing rule in one or two sentences — what must (or must not) happen, every time, going forward. Write it as an ongoing directive, not a one-off decision. If the rule carries a specific number or limit, state that number here in plain words.>

## Scope

<Say where and when this rule applies, and — just as important — where it does not, in one to three sentences. Name the situations it governs so a reader can tell whether a given case is covered.>

## Rationale

<Explain, in plain language a non-engineer can follow, why this rule exists and what it protects against — informed, not alarmed. Two to four sentences, no internal jargon. If the rule carries a tunable number, say in plain terms when you would raise it and when you would lower it.>

## Enforcement-tier

<State honestly how this rule is held: a posture (an expectation people are trusted to follow), a soft warning (a non-blocking nudge), a hard check that blocks the work, or a layered combination — and name which part is which. Never describe an expectation as if a machine guaranteed it. If a check enforces only part of the rule, say plainly what it does and does not catch.>
