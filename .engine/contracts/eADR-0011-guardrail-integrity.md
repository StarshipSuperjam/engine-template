title: Guardrail integrity — the builder cannot silently weaken its own enforcement

## Decision

The builder runs inside the very repository whose enforcement protects it, so it can reach its own guardrails — the permission and enforcement config, the CI workflows and check definitions, CODEOWNERS, and the branch ruleset. It may *change* them, since a legitimate install can strengthen them, but it may never *weaken* them silently. A weakening change — removing, disabling, renaming, or loosening a gate, or editing a ruleset-affecting file — is a governance-critical event: it hard-blocks the merge until the operator performs one deliberate affirmative act distinct from the ordinary merge click (a checkbox or an applied label, never a typed phrase or a command to run), and that prompt names in plain language exactly which protection weakened and what the AI could then do unwatched. A pure strengthening passes with no alarm. The guard that detects weakening is itself a gate — bound into the ruleset by a frozen name so deleting or renaming it deadlocks the merge, and evaluated from a trusted source that reads only the diff and never runs the proposed code, so a change can never disarm the guard meant to catch it.

## Significance

This establishes the engine's one irreducible promise: the checks the operator trusts cannot be turned off behind their back. Every later gate, check, and review the operator relies on inherits its credibility from this — without it, the whole "walk away and trust the evidence" bargain collapses, because a builder could disarm the evidence and report success. Two disciplines fall under this law and must be respected, not re-opened as separate ones. First, what counts as a guarded file is defined by a *property*, not a fixed path list: a guarded file is one that constitutes or configures an enforcement gate, realized as a permanent floor plus every check rule's enforcement script discovered by presence — so a future check is guarded by merely existing, never by being remembered on a hand-list. When in doubt a file is treated as guarded (fail-safe), because a wrongly-excluded gate is a silently-weakenable one, an unrecoverable loss, whereas over-inclusion only costs an extra acknowledgment. Non-gate tooling — boot, memory, telemetry, the self-review renderer — is excluded because it is neither a gate nor floor, not by an exclusion list. Second, any generated artifact this enforcement produces carries its warrant at the point of use: what a green result shows, what it does not show, and what review still covers. A green guard proves the diff was classified and acknowledged, never that the change is wise; later work must state that bound where the operator meets the result, never let the artifact read as authority it lacks.

## Rationale

The operator approves on evidence and prose, never by reading code, so the burden of trustworthiness rests entirely on the engine — and the builder lives inside the repo whose enforcement is the only thing making its work checkable. An unguarded builder could quietly disarm exactly the checks that earn the operator's absence, the worst-case betrayal. The hard merge block is warranted because a system's own gates are a governance-critical invariant where mechanical refusal, not a judgment call, is the right tier. The acknowledgment is one deliberate act and no more — a bare merge click is not informed consent, but a typed phrase is a mistype trap for a non-engineer and a ceremony gauntlet trains rubber-stamping. The trade-off is named honestly by closure tier: solo reduces the threat to *cannot weaken silently* (the operator still holds the credentials, so the engine could produce the acknowledgment itself — the gesture makes weakening deliberate and visible, not impossible), while only a distinct enforcement-admin identity in a team makes it *cannot weaken at all*. The design never dresses the solo gate as airtight prevention.

## Anti-choice

The strongest rejected alternative was a guard that simply *surfaces* every enforcement-config change for the operator to notice. It was rejected because undifferentiated alarms destroy the signal precisely when it matters: firing on every benign install — which legitimately strengthens enforcement — trains the operator to rubber-stamp, so the one genuinely dangerous weakening slides through unread. Discriminating weakening from strengthening, and hard-blocking only the former, is what keeps the alarm meaningful. Two narrower alternatives lost for the same family of reasons: a standing local pre-write block on guardrail files (an always-on gate orthogonal to any work stance, with no natural owner, that the platform does not reliably honor for every edit path, and that the operator never sees — so it adds no guarantee the merge gate does not already give), and recording the gap as a concrete list of blocked file paths (its real content is a *law* about enforcement integrity belonging at the trust seam, and a hand-list silently rots as new gates appear). Evaluating the guard against the proposed change's own code was also rejected: the guarding change could then neuter the very guard meant to catch it.
