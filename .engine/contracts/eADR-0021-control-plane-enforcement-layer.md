## Decision

The GitHub layer that makes every other guardrail real is fixed as a set of **contracts and laws, never as the scripts or workflow files that satisfy them**. It fixes: one **branch ruleset** (protection rules and required-check bindings are a single object) carrying an enumerated **protection floor** — a pull request before merging (zero required approvals in solo so a sole owner can still merge, one code-owner review in team), status checks passing, conversation resolution, blocked force-pushes, restricted deletion — with bypass named honestly (solo can bypass; team cannot). It establishes a **bootstrap contract** (the ruleset is a setting, not a file, so it is applied once by an operator-privileged actor — never the Actions token — behind a committed guard that fails loud and surfaces the unprotected state in plain words). It settles an **honest identity model** (solo: the AI commits as the operator and the merge click is *informed consent, not code review*; team: a distinct AI identity makes the operator the enforced reviewer). It fixes a **pull-request section contract** with a gated completeness check, **human issue templates** that guide but never gate, an **engine-domain label scheme** with one presence-based label owned here and read-only for every consumer, a **CI-harness seam**, a **single-flight cron law**, and a visibility-scaled **security floor**. Wherever a neighbour owns the leaf — the ruleset payload, the bootstrap mechanism, check content, suite membership, label strings — this layer fixes the invariant and defers the implementation.

## Significance

This establishes that **trust is routed through automated, operator-weighable enforcement at the merge, not through code the operator cannot read** — the merge click is consent on evidence, never a review pretense. Everything downstream depends on this gate, so it is settled first: any later system that touches branch protection, identity, the pull-request shape, issue creation, scheduled work, or scanning must adapt to these contracts rather than redefine them. The protection floor is a fixed minimum — removing or loosening any rule is a weakening-class change that must trip the integrity guard and clear the operator's distinct acknowledgment; it is never an ordinary edit. The engine-domain label is the single routing substrate the self-monitoring loop runs over: producers apply it, every consumer only reads it. Because this layer locks contracts and not leaves, neighbours retain their implementation lanes — the ruleset payload, bootstrap UX, check wiring, and label strings stay theirs to build, bound by the invariants fixed here. The pull-request body whose completeness this layer gates is the single change-history fixed by eADR-0014 (one history; living documents everywhere else): that record owns the history, this layer owns the merge-time gate over it — not two owners of the same record.

## Rationale

The control plane is the gate every other guardrail sits downstream of, so it had to be settled early — but it abuts unsettled neighbours (provisioning, audits, validation, the module system). Fixing **contracts rather than concrete files** resolves the tension: the layer can be established now without front-running who runs bootstrap, which audit cron exists, what a check contains, or which checks are in the suite. The operator is a non-engineer who cannot review code, so the gate's strength is the only honest source of trust; naming the solo merge as *consent* and making the pull request structurally complete is what keeps that consent informed rather than theatre. The protection floor had to be enumerated, not merely asserted, because a gate whose rules are unwritten cannot be built or trusted — and each omitted rule (branches-up-to-date, signed commits, linear history, restrict-updates) is excluded for a reason that protects the non-engineer's merge-and-walk flow, not by oversight. The engine-domain label had to be owned by a foundation, not an optional module, because required-core systems read it whether or not that module is installed.

## Anti-choice

The strongest rejected alternative was to **lock the enforcement layer as concrete leaves** — a specific bootstrap script and trigger, a concrete audit cron file, the exact ruleset payload — so a cold builder could construct it directly. This was rejected because those leaves front-run systems that are not yet settled: the bootstrap mechanism and first-run experience belong to provisioning, the audit schedule to the audits system, check content to validation, and suite membership to the module system. Pinning them here would force a refactor of this gate every time a neighbour firms up, and would re-source the same rule in two places. Fixing the invariant and deferring the implementation lets this layer ratify early and stay stable while its neighbours fill their own lanes. A second rejected alternative was to call the solo merge click "human review" — rejected because a non-engineer cannot review code, so dressing the click as review would be advisory theatre and would corrupt the trust story the whole layer exists to make honest.

## Status

accepted
